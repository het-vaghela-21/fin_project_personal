7. AUTHENTICATION FLOW (DETAILED)
================================================================================

  STEP 1: User submits login form
    → Firebase signInWithEmailAndPassword(auth, email, password)
    → Firebase verifies credentials server-side
    → Returns: UserCredential object

  STEP 2: syncToMongo(payload)
    → POST /api/users/sync
    → MongoDB: findOneAndUpdate({uid}, {...userData}, {upsert: true})
    → Returns role from DB
    → Special case: admin@finai.com → instantly returns "admin"

  STEP 3: setAuthCookies(token, role)
    → document.cookie = "auth-token=<Firebase JWT>; max-age=3600"
    → document.cookie = "user-role=<role>; max-age=3600"
    → These cookies are read by middleware.ts on EVERY page request

  STEP 4: recordLoginEvent(uid) [non-blocking]
    → fetch("https://api.ipify.org") → get public IP address
    → POST /api/users/login-event {uid, ip, device}
    → Server: AbortController timeout (2 seconds) on geo-lookup
    → Server: fire-and-forget MongoDB write (w:0, no ack needed)
    → Response returns instantly (<200ms)

  STEP 5: router.push()
    → role === "admin"  →  /admin
    → role === "user"   →  /dashboard

  MIDDLEWARE (runs on every request to /dashboard/* and /admin/*):
    → Reads "auth-token" cookie
    → No token? → redirect /auth/login?redirect=<original-path>
    → Has token on /auth page? → redirect to appropriate home

8. DATA FLOW INSIDE DASHBOARD
================================================================================

  User navigates to ANY dashboard page
              |
              v
  DashboardLayout.tsx renders:
    1. DashboardBackground (memoized — renders ONCE EVER)
    2. DashboardProvider (mounts ONCE per session)
              |
              v
  DashboardProvider useEffect fires (on user login):
    → Parallel fetch (Promise.all):
        GET /api/transactions  →  MongoDB: find({userId}).sort({date:-1})
        GET /api/goals         →  MongoDB: find({userId}).sort({createdAt:-1})
    → Both use compound/field indexes → fast even with many records
    → State: transactions[], goals[]
              |
              v
  Any sub-page calls useDashboard():
    → Gets transactions, goals from context
    → NO re-fetching on navigation
    → Changes (add/delete) update state locally + hit API
    → Single source of truth for entire dashboard

  Data update flow (e.g. add transaction):
    User submits form
      → addTransaction(tx) in DashboardProvider
      → POST /api/transactions {amount, type, category, title, date}
      → MongoDB creates document
      → Response: new transaction object
      → setTransactions([newTx, ...prev]) — instant UI update
